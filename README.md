# Candidate Ranking Pipeline

A scoring system that ranks job candidates against a fixed job description (JD) using a blend of **semantic embedding similarity**, **keyword skill matching**, **recruiting-platform behavior signals**, and a **resume-honesty penalty**. The pipeline reads a JSONL file of candidate profiles, scores and ranks all of them, and writes the top 100 to a CSV with a short auto-generated explanation for each.

---

## How it works (pipeline overview)

```
candidates.jsonl
      │
      ▼
 ┌────────────────────┐ career text, profile text       ┌────────────────┐
 │ src/helper/embed.py│ ───────────────────────────────► │ *.npy embeddings│
 └────────────────────┘ (SentenceTransformer, batched)   └────────────────┘
                                                              │
                                                              ▼
candidates.jsonl ──► src/pipeline.py ──► src/helper/score.py (per candidate)
                          │              ├─ score_embeddings()   → career_score, profile_score
                          │              ├─ score_skill_keyword()→ skill_score, matched_skills
                          │              ├─ score_behavior()     → behaviour_score
                          │              └─ honeypot_penalty()   → penalty
                          │
                          ├─ final_score = 0.70·career_score + 0.20·behaviour_score
                          │                + 0.10·profile_score + 0.10·skill_score − penalty
                          │
                          ├─ src/helper/reasons.py → relevant_field() → human-readable reasoning string
                          │
                          ▼
                  submission/submissions.csv (top 100, ranked)
```

`src/helper/embed.py` is a **one-time (or per-refresh) precompute step**. `src/pipeline.py` is the **scoring/ranking step** that runs against the precomputed embeddings. They are separate scripts because embedding generation is the slow part (model inference over every candidate) and doesn't need to be repeated every time you tweak scoring weights.

## Setup

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Prepare your data

Place a `candidates.jsonl` file in `data/`. Each line must be a JSON object with this shape:

```json
{
  "candidate_id": "c_00123",
  "profile": {
    "headline": "Senior ML Engineer",
    "summary": "Built large-scale retrieval systems...",
    "years_of_experience": 6.5
  },
  "skills": [
    {"name": "Vector Search", "proficiency": "expert"},
    {"name": "Python", "proficiency": "advanced"}
  ],
  "career_history": [
    {
      "title": "Senior ML Engineer",
      "industry": "Tech",
      "description": "Owned the candidate-matching ranking system...",
      "start_date": "2021-03-01",
      "end_date": null
    }
  ],
  "redrob_signals": {
    "open_to_work_flag": true,
    "recruiter_response_rate": 0.8,
    "notice_period_days": 30,
    "last_active_date": "2026-06-10",
    "interview_completion_rate": 0.9,
    "saved_by_recruiters_30d": 4,
    "github_activity_score": 65
  }
}
```

All fields shown above are read somewhere in the pipeline, so every candidate record needs all of them (`end_date: null` is handled and treated as "currently employed").

### 3. Set the target job description

`src/helper/profile_set.py` hardcodes the job description as two free-text blocks (`jd_career_text`, `jd_profile_text`) and the keyword skill lists (`required_skills`, `preffered_skills`). **This file is the only place you need to edit to re-target the pipeline at a different role.** It also defines `reason_templates`, the category labels (e.g. "retrieval and ranking", "llm", "mlops") used to describe each candidate's strongest field in the generated reasoning text.

Note: every change here that touches `jd_career_embedding` or `jd_profile_embedding` re-embeds the JD text on import (cheap, single string) — but changes to `required_skills` / `preffered_skills` don't require re-running `src/helper/embed.py`, since those are matched as plain keywords against candidate skill lists, not embedded.

## Running the pipeline

From the project root (`Smart recruiter/`):

```bash
# Step 1 — generate embeddings for every candidate (run once, or whenever candidates.jsonl changes)
python src/helper/embed.py

# Step 2 — score, rank, and explain candidates
python src/pipeline.py
```

`src/pipeline.py` writes `submission/submissions.csv` with columns:

| column | description |
|---|---|
| `candidate_id` | candidate's ID from the input data |
| `rank` | 1–100, 1 = best match |
| `score` | final blended score (see formula below) |
| `reasoning` | one auto-generated sentence explaining the match |

> **Note:** `src/scan.py` and `submission/submission_metadata.yaml` are also part of this project but aren't documented here yet — see their source for current behavior.

## Scoring formula

For each candidate:

```
final_score = 0.70 × career_score
            + 0.20 × behaviour_score
            + 0.10 × profile_score
            + 0.10 × skill_score
            − honeypot_penalty
```

### `career_score` / `profile_score` — semantic similarity (`score_embeddings`)
Cosine similarity (dot product, since all embeddings are L2-normalized) between:
- the candidate's **career history text** (titles + industries + descriptions, concatenated) and the JD's career-focused embedding, and
- the candidate's **profile text** (headline + summary + skills) and the JD's profile-focused embedding.

Both use `sentence-transformers/all-MiniLM-L6-v2`.

### `skill_score` — keyword matching (`score_skill_keyword`)
Each candidate skill is normalized (lowercased, punctuation-normalized) and checked against the `required_skills` and `preffered_skills` sets. Matched skills are weighted by proficiency (`expert`=1.0, `advanced`=0.8, `intermediate`=0.5, else 0.3), summed, then **divided by the total number of required/preferred skills** — so this measures *coverage of the skill list*, not just average proficiency. Final skill score: `0.8 × required_coverage + 0.2 × preferred_coverage`.

### `behaviour_score` — recruiting signal blend (`score_behavior`)
A weighted sum of platform engagement signals (open-to-work flag, recruiter response rate, days since last active, notice period, interview completion rate, recently saved-by-recruiters count, GitHub activity):

```
0.30·open_to_work + 0.25·response_rate + 0.15·recency
+ 0.10·notice_period + 0.10·interview_completion
+ 0.05·saved_by_recruiters + 0.05·github_activity
```

### `honeypot_penalty` — resume-honesty check (`honeypot_penalty`)
Two independent red flags, additive:
- **+0.2** if the candidate's claimed `years_of_experience` is more than 1.5× what their actual `career_history` date ranges support (claims far more tenure than their listed jobs cover).
- **+0.4** if `skill_score > 0.5` but `career_score < 0.3` — i.e., their listed skills look like a strong keyword match, but their actual career narrative doesn't semantically resemble the JD at all (a classic sign of keyword-stuffed skills with no real matching experience).

### Auto-generated reasoning (`relevant_field`, in `reasons.py`)
For each top-100 candidate, builds one sentence using:
- years of experience + most recent job title,
- their **strongest semantic field** (computed by comparing their career embedding against the `reason_templates` category embeddings from `profile_set.py` and taking the closest match),
- strengths (skill count ≥ 6, career_score > 0.60, behaviour_score > 0.70, top 1–2 matched skills),
- and a single concern if one applies (notice period ≥ 90 days, or behaviour_score < 0.4).

## Extending this project

- **Re-targeting a new role:** edit only `src/helper/profile_set.py` (JD text, skill sets, reason templates).
- **Changing the score blend:** edit the `final_score` formula in `src/pipeline.py`.
- **Adding a new behavior signal:** add it to `redrob_signals` in your data and to the weighted sum in `score_behavior()` (keep weights summing to 1.0 if you want `behaviour_score` to stay in `[0, 1]`).
- **Swapping the embedding model:** change the model string in both `src/helper/embed.py` and `src/helper/profile_set.py` — they must match, since mixing embeddings from two different models will produce meaningless similarity scores. After swapping, re-run `src/helper/embed.py` to regenerate all `.npy` files.
