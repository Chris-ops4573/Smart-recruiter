# Candidate Ranking Pipeline

A scoring system that ranks job candidates against a fixed job description (JD) using a blend of **semantic embedding similarity**, **keyword skill matching**, **recruiting-platform behavior signals**, and a **resume-honesty penalty**. The pipeline reads a JSONL file of candidate profiles, scores and ranks all of them, and writes the top candidates to a CSV file with a short auto-generated explanation for each. A separate converter script can turn that CSV into an `.xlsx` workbook for submission.

There are two ways to run this:
- **CLI** (`src/pipeline.py` + `src/convert.py`) — for re-running locally, tweaking weights, etc.
- **Streamlit app** (`src/app.py`) — a simple upload/run/download UI, also shipped as a Docker image.

---

## How it works (pipeline overview)

```
candidates.jsonl
      │
      ▼
 ┌────────────────────┐ career text, profile text        ┌────────────────┐
 │ src/helper/embed.py│ ───────────────────────────────► │*.npy embeddings│
 └────────────────────┘ (SentenceTransformer, batched)   └────────────────┘
                                                              │
                                                              ▼
candidates.jsonl ──► src/pipeline.py ──► src/helper/score.py (per candidate)
                          │              ├─ score_embeddings()   → career_score, profile_score
                          │              ├─ score_skill_keyword()→ skill_score, matched_skills
                          │              ├─ score_behavior()     → behaviour_score
                          │              └─ honeypot_penalty()   → penalty
                          │
                          ├─ final_score = 0.70·career_score + 0.20·behaviour_score
                          │                + 0.10·profile_score + 0.10·skill_score − penalty
                          │
                          ├─ src/helper/reasons.py → relevant_field() → human-readable reasoning string
                          │
                          ▼
                  <output CSV path, given as CLI arg> (ranked, top N)
                          │
                          ▼ (optional)
                  src/convert.py  →  .xlsx workbook
```

`src/helper/embed.py` is a **one-time (or per-refresh) precompute step**. `src/pipeline.py` is the **scoring/ranking step** that runs against the precomputed embeddings. They are separate scripts because embedding generation is the slow part (model inference over every candidate) and doesn't need to be repeated every time you tweak scoring weights.

`src/pipeline.py` writes **CSV**, not Excel directly. If you need an `.xlsx` deliverable, run `src/convert.py` afterward to convert it (see [Running the pipeline](#running-the-pipeline)).

The Streamlit app (`src/app.py`) wraps the same `embed()` and `rank_candidates()` functions behind a simple upload-and-download UI and writes CSV only — it does not call `convert.py`.

## Required directory structure

The code uses **relative paths** (`../data/...`, `../embeddings/...`) and a Python package import (`from helper.score import ...`), so the on-disk layout is not optional — it must look like this:

```
Smart recruiter/                  # project root
├── .venv/
├── data/
│   └── candidates.jsonl          # input: one JSON object per line
├── embeddings/
│   ├── career.npy                 # generated by src/helper/embed.py
│   ├── profile.npy                # generated by src/helper/embed.py
│   ├── jd_career.npy              # generated by src/helper/embed.py
│   ├── jd_profile.npy             # generated by src/helper/embed.py
│   └── reason.npy                 # generated by src/helper/embed.py
├── src/
│   ├── helper/
│   │   ├── __init__.py           # empty file — required for `helper` to be importable as a package
│   │   ├── embed.py
│   │   ├── profile_set.py
│   │   ├── reasons.py
│   │   └── score.py
│   ├── app.py                    # Streamlit UI (upload → rank → download CSV)
│   ├── convert.py                # CSV → XLSX converter (helper script, not part of scoring)
│   ├── pipeline.py
│   └── scan.py
├── submission/
│   └── (CSV/XLSX output lands here, depending on how you invoke the pipeline)
├── .dockerignore
├── .gitignore
├── Dockerfile                     # builds the Streamlit app, see "Running via Docker" below
├── README.md
└── requirements.txt
```

> Note: `embeddings/` is the directory you pass as the last argument to the **CLI** flow (`python src/pipeline.py ... ../embeddings`). The **Streamlit app** generates its own embeddings into a temporary `embeddings_temp/` directory per run and deletes it afterward — it does not read from or write to `embeddings/`.

## Setup (IGNORE for redrob AI hackathon judges)

### 1. Install dependencies

```bash
pip install -r requirements.txt
```

### 2. Prepare your data

Place a `candidates.jsonl` file in `data/`. Each line must be a JSON object with this shape:

```json
{
  "candidate_id": "c_00123",
  "profile": {
    "headline": "Senior ML Engineer",
    "current_title": "Senior ML Engineer",
    "summary": "Built large-scale retrieval systems...",
    "years_of_experience": 6.5
  },
  "skills": [
    {"name": "Vector Search", "proficiency": "expert"},
    {"name": "Python", "proficiency": "advanced"}
  ],
  "career_history": [
    {
      "title": "Senior ML Engineer",
      "company": "Acme Corp",
      "industry": "Tech",
      "description": "Owned the candidate-matching ranking system...",
      "start_date": "2021-03-01",
      "end_date": null
    }
  ],
  "redrob_signals": {
    "open_to_work_flag": true,
    "recruiter_response_rate": 0.8,
    "notice_period_days": 30,
    "last_active_date": "2026-06-10",
    "interview_completion_rate": 0.9,
    "saved_by_recruiters_30d": 4,
    "github_activity_score": 65,
    "skill_assessment_scores": {
      "Vector Search": 88.0
    },
    "applications_submitted_30d": 5,
    "willing_to_relocate": true
  }
}
```

All fields shown above are read somewhere in the pipeline, so every candidate record needs all of them (`end_date: null` is handled and treated as "currently employed"). `company` and `current_title` are only used to build the text fed into the embedding model (`src/helper/embed.py`) — they don't directly affect scores beyond that. `skill_assessment_scores`, `applications_submitted_30d`, and `willing_to_relocate` feed the auto-generated reasoning text (see below) rather than the score itself.

### 3. Set the target job description

`src/helper/profile_set.py` hardcodes the job description as two free-text blocks (`jd_career_text`, `jd_profile_text`) and the keyword skill lists (`required_skills`, `preffered_skills`). **This file is the only place you need to edit to re-target the pipeline at a different role.** It also defines `reason_templates`, the category labels (e.g. "retrieval and ranking", "llm", "mlops") used to describe each candidate's strongest field in the generated reasoning text.

Note: every change here that touches `jd_career_embedding` or `jd_profile_embedding` re-embeds the JD text on import (cheap, single string) — but changes to `required_skills` / `preffered_skills` don't require re-running `src/helper/embed.py`, since those are matched as plain keywords against candidate skill lists, not embedded.

## Running the pipeline

### Option A — CLI

From the project root (`Smart recruiter/`):

```bash
# Step 1 — generate embeddings for every candidate (run once, or whenever candidates.jsonl changes)
# args: <input candidates.jsonl> <output embeddings directory>
python src/helper/embed.py /data/candidates.jsonl embeddings

# Step 2 — score, rank, and explain candidates
# args: <number of candidates to rank> <input candidates.jsonl> <output .csv path> <embeddings directory>
python src/pipeline.py 100 data/candidates.jsonl submission/bge_submissions.csv embeddings

# Step 3 (optional) — convert the CSV to .xlsx if you need a workbook deliverable
python src/convert.py submission/bge_submissions.csv submission/bge_submissions.xlsx
```

`src/pipeline.py` takes four positional CLI arguments, in this order: `<candidate_count> <candidate_file> <output_csv_path> <embeddings_dir>`. It writes the **CSV** file at `output_csv_path` with columns:

| column | description |
|---|---|
| `candidate_id` | candidate's ID from the input data |
| `rank` | 1–N, 1 = best match |
| `score` | final blended score (see formula below) |
| `reasoning` | one auto-generated sentence explaining the match |

If `candidate_count` is larger than the number of candidates in the input file, the script prints an error and exits without writing output.

`src/convert.py` is a thin helper (`pandas.read_csv` → `pandas.to_excel`) that takes an input CSV path and output XLSX path as its two CLI arguments — it does no scoring, it's purely a format conversion.

### Option B — Streamlit app

From `src/`:

```bash
streamlit run app.py
```

This opens a browser UI where you upload a `candidates.jsonl` file, pick how many candidates to rank, and download a CSV of results. It generates its own embeddings on the fly into a temporary directory (cleaned up after the run), so there's no separate "precompute" step to run first — at the cost of being slower per-run than the CLI flow for repeated experiments, since it re-embeds every time it's launched fresh.

### Running via Docker (for hackathon judges)

A prebuilt image is published to Docker Hub. To run it without setting up a Python environment:

```bash
docker pull christiner4573/smart-recruiter:latest
docker run -p 8501:8501 christiner4573/smart-recruiter:latest
```

Then open `http://localhost:8501` in your browser — this serves the same Streamlit app described in Option B above.

## Scoring formula

For each candidate:

```
final_score = 0.70 × career_score
            + 0.20 × behaviour_score
            + 0.10 × profile_score
            + 0.10 × skill_score
            − honeypot_penalty
```

### `career_score` / `profile_score` — semantic similarity (`score_embeddings`)
Cosine similarity (dot product, since all embeddings are L2-normalized) between:
- the candidate's **career history text** (titles + industries + descriptions, concatenated) and the JD's career-focused embedding, and
- the candidate's **profile text** (headline + summary + skills) and the JD's profile-focused embedding.

Both use `BAAI/bge-base-en-v1.5`.

### `skill_score` — keyword matching (`score_skill_keyword`)
Each candidate skill is normalized (lowercased, punctuation-normalized) and checked against the `required_skills` and `preffered_skills` sets. Matched skills are weighted by proficiency (`expert`=1.0, `advanced`=0.8, `intermediate`=0.5, else 0.3), summed, then **divided by the total number of required/preferred skills** — so this measures *coverage of the skill list*, not just average proficiency. Final skill score: `0.8 × required_coverage + 0.2 × preferred_coverage`.

### `behaviour_score` — recruiting signal blend (`score_behavior`)
A weighted sum of platform engagement signals (open-to-work flag, recruiter response rate, days since last active, notice period, interview completion rate, recently saved-by-recruiters count, GitHub activity):

```
0.30·open_to_work + 0.25·response_rate + 0.15·recency
+ 0.10·notice_period + 0.10·interview_completion
+ 0.05·saved_by_recruiters + 0.05·github_activity
```

### `honeypot_penalty` — resume-honesty check (`honeypot_penalty`)
Two independent red flags, additive:
- **+0.2** if the candidate's claimed `years_of_experience` is more than 1.5× what their actual `career_history` date ranges support (claims far more tenure than their listed jobs cover).
- **+0.4** if `skill_score > 0.5` but `career_score < 0.3` — i.e., their listed skills look like a strong keyword match, but their actual career narrative doesn't semantically resemble the JD at all (a classic sign of keyword-stuffed skills with no real matching experience).

### Auto-generated reasoning (`relevant_field`, in `reasons.py`)
For each ranked candidate in the output, builds one sentence using:
- years of experience + most recent job title,
- their **strongest semantic field** (computed by comparing their career embedding against the `reason_templates` category embeddings from `profile_set.py` and taking the closest match),
- strengths — skill count ≥ 6, career_score > 0.60, a verified `skill_assessment_scores` entry ≥ 70, recruiter-engagement detail when behaviour_score > 0.70 (response rate or interview completion rate), long stable tenure in the current role, and the top 1–2 matched skills,
- and concerns — notice period ≥ 90 days, behaviour_score < 0.4, short average tenure across past roles, very high recent application volume, not flagged open to work, or unwillingness to relocate (only the first applicable concern is shown).

## Extending this project

- **Re-targeting a new role:** edit only `src/helper/profile_set.py` (JD text, skill sets, reason templates).
- **Changing the score blend:** edit the `final_score` formula in `src/pipeline.py`. Note this formula is currently duplicated inline in `app.py`'s call to `rank_candidates`, not just in `pipeline.py` — keep them in sync, or refactor `app.py` to call the same code path.
- **Adding a new behavior signal:** add it to `redrob_signals` in your data and to the weighted sum in `score_behavior()` (keep weights summing to 1.0 if you want `behaviour_score` to stay in `[0, 1]`).
- **Swapping the embedding model:** change the model string in both `src/helper/embed.py` and `src/helper/profile_set.py` — they must match, since mixing embeddings from two different models will produce meaningless similarity scores. After swapping, re-run `python src/helper/embed.py <candidates.jsonl> <embeddings dir>` to regenerate all `.npy` files (CLI flow) — the Streamlit app picks up the change automatically on its next run since it re-embeds every time.
- **`src/scan.py`** exists in this repo but isn't currently documented here — if you're extending it, add a section describing what it does.
